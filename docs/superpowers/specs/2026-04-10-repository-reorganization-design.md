# Repository Reorganization Design

**Date:** 2026-04-10
**Status:** Approved
**Author:** Claude Code

## Overview

Reorganize the personal website repository to clearly separate editable content files from infrastructure, generated files, and static assets. The goal is to make it immediately obvious which files need to be edited to update the website.

## Current State

Files are scattered at the root level:
- Source content (`index.md`, `translations.js`)
- Generated output (`index.html`)
- Stylesheets (`reset.css`, `index.css`)
- Scripts (`index.js`, `i18n.js`)
- Build template (`template.html`)
- Static assets (`castle.jpg`)
- Configuration files (`Makefile`, `flake.nix`, etc.)
- PDFs (`files/` directory - already well-organized)

This makes it unclear which files are the "source of truth" for content updates.

## User Requirements

From discovery questions:
1. Primary workflow: Edit markdown content (`index.md`) and translations (`translations.js`)
2. PDFs updated occasionally - can be in separate location
3. Generated files should be in a build/output directory
4. Configuration files: mixed approach - some at root (Makefile), template moved with web assets
5. Deployment via GitHub Actions with automated builds

### Rationale

**content/** directory:
- Contains only the two files edited most frequently
- Clear workspace for content updates
- Minimal cognitive load - no need to ignore infrastructure files

**assets/** directory:
- Groups all web infrastructure by type (styles, scripts, images)
- Subdirectories provide clear organization
- Template lives with other web assets
- Rarely modified, so nested structure is acceptable

**build/** directory:
- Self-contained website output
- Can be deployed as-is to any static host
- Excluded from git (generated by CI)
- Contains all necessary files with correct paths

**Root level**:
- Only essential configuration and documentation
- Makefile stays accessible for local builds
- Nix flake files for development environment
- Git and deployment configuration

## Build Process

### Updated Makefile

The Makefile will implement these targets:

**all** (default):
1. Generate `build/index.html` from `content/index.md`
2. Copy all assets to `build/`
3. Copy `files/` to `build/files/`
4. Copy `CNAME` to `build/`

**clean**:
- Remove entire `build/` directory

**serve** (optional):
- Start local HTTP server in `build/` for preview

### Pandoc Build Command

```bash
pandoc --toc -s \
  --css styles/reset.css \
  --css styles/index.css \
  -i content/index.md \
  -o build/index.html \
  --template=assets/template.html
```

Key changes from current:
- Input: `content/index.md` (was `index.md`)
- Output: `build/index.html` (was `index.html`)
- Template: `assets/template.html` (was `template.html`)
- CSS paths: `styles/*.css` (relative to build output)

### Asset Copying

All assets copied to maintain working relative paths:
- `assets/styles/*.css` в†’ `build/styles/`
- `assets/scripts/*.js` в†’ `build/scripts/`
- `assets/images/*` в†’ `build/images/`
- `files/` в†’ `build/files/` (recursive)
- `CNAME` в†’ `build/CNAME`

### Template Updates

`assets/template.html` needs path updates:
- CSS: `reset.css` в†’ `styles/reset.css`
- CSS: `index.css` в†’ `styles/index.css`
- Script tags need similar updates if they reference files

## Git & Version Control

### .gitignore Updates

Add:
```
build/
```

This excludes all generated files from version control.

### File Movements

Use `git mv` to preserve history:
```bash
git mv index.md content/
git mv translations.js content/
git mv reset.css assets/styles/
git mv index.css assets/styles/
git mv index.js assets/scripts/
git mv i18n.js assets/scripts/
git mv template.html assets/
git mv castle.jpg assets/images/
```

Remove old generated file:
```bash
git rm index.html
```

## Deployment: GitHub Actions

### Workflow Configuration

New file: `.github/workflows/deploy.yml`

**Trigger:** Push to `main` branch

**Jobs:**
1. **Setup environment:**
   - Checkout repository
   - Install Nix (respects existing `flake.nix`)
   - Or: Install Pandoc directly if simpler

2. **Build:**
   - Run `make` to generate site in `build/`

3. **Deploy:**
   - Use `actions/deploy-pages@v2` or similar
   - Deploy contents of `build/` to GitHub Pages

### GitHub Pages Configuration

Repository settings:
- Pages source: "GitHub Actions" (not "Deploy from a branch")
- This allows the workflow to handle deployment

### Developer Workflow

**To update content:**
1. Edit `content/index.md` or `content/translations.js`
2. (Optional) Run `make` locally to preview in `build/index.html`
3. Commit changes: `git add content/ && git commit -m "Update content"`
4. Push: `git push`
5. GitHub Actions builds and deploys automatically

**Local preview:**
- `build/` directory exists locally after running `make`
- Open `build/index.html` in browser to preview
- `build/` is never committed (excluded by .gitignore)

## Migration Plan

### Phase 1: Structure Setup
1. Create new directories: `content/`, `assets/styles/`, `assets/scripts/`, `assets/images/`, `build/`
2. Create `.github/workflows/` directory

### Phase 2: Move Files
1. Use `git mv` for all file movements (preserves history)
2. Move content files to `content/`
3. Move assets to appropriate `assets/` subdirectories

### Phase 3: Update Configuration
1. Update `Makefile` with new paths and asset copying
2. Update `assets/template.html` with correct CSS/JS paths
3. Update `.gitignore` to exclude `build/`
4. Create `.github/workflows/deploy.yml`

### Phase 4: Testing
1. Run `make clean && make`
2. Verify `build/index.html` generates correctly
3. Check all assets copied to `build/`
4. Verify all links work (especially PDF links)
5. Open `build/index.html` in browser

### Phase 5: Cleanup
1. Remove old `index.html` from root: `git rm index.html`
2. Verify no stray files remain at root

### Phase 6: Commit & Deploy
1. Commit reorganization in single commit with clear message
2. Push to GitHub
3. Verify GitHub Actions workflow runs successfully
4. Check deployed site at GitHub Pages URL

## Testing & Verification

### Pre-deployment Checklist
- [ ] `make clean && make` succeeds without errors
- [ ] `build/index.html` exists and opens in browser
- [ ] All CSS styles load correctly
- [ ] All JavaScript functionality works
- [ ] PDF links work (`files/` directory accessible)
- [ ] Translation switching works
- [ ] Images display correctly
- [ ] Navigation works

### Post-deployment Checklist
- [ ] GitHub Actions workflow completes successfully
- [ ] Site deploys to GitHub Pages
- [ ] Custom domain still works (CNAME)
- [ ] All pages and assets load
- [ ] No 404 errors in browser console

## Rollback Plan

If issues arise:
1. Revert the reorganization commit: `git revert <commit-hash>`
2. Push revert
3. Old structure restored, site works as before

## Benefits

1. **Clarity:** Immediately obvious which files to edit (`content/` directory)
2. **Organization:** Related files grouped logically (styles together, scripts together)
3. **Clean git:** No generated files in version control
4. **Automation:** GitHub Actions handles building and deployment
5. **Safety:** `build/` regenerated from source, never manually edited
6. **Scalability:** Easy to add more content files, assets, or pages in the future

## Trade-offs

1. **Initial effort:** One-time reorganization and configuration
2. **Path updates:** Makefile and template need path updates
3. **Learning curve:** Minor adjustment to new structure
4. **CI dependency:** Site builds happen on GitHub (still can build locally)

The benefits significantly outweigh the one-time migration effort.
